# Render raw HTML in notes as literal text

> Roadmap item (Deferred): "Render raw HTML as text: embedded HTML in a note renders as literal
> characters, not as HTML elements." Its own small PR. **Prerequisite for
> [interactive-task-lists.md](./interactive-task-lists.md).**

## Why?

Note bodies are Markdown, rendered by `src/lib/markdown/render.ts` (`marked` → DOMPurify). Today
`marked` **passes raw HTML through**: if a note contains `<b>hi</b>`, `<img src=x>`, or
`<input type="checkbox">`, those become real DOM elements in View mode (DOMPurify only strips the
*dangerous* bits — `<script>`, `on*` handlers, `javascript:` URLs — the benign tags still render).

Two problems with that for a personal Markdown notes tool:

1. **Unpredictable rendering.** A user typing an HTML-looking snippet (e.g. pasting a code fragment
   without fencing it) sees it silently interpreted as markup instead of shown as text.
2. **Larger attack surface than necessary.** Rendering arbitrary (even sanitized) HTML from synced,
   effectively-untrusted content is more than a notes app needs. Escaping raw HTML is defense-in-depth
   *on top of* DOMPurify.

It also unblocks interactive task lists: once raw HTML is text, **every** checkbox rendered in View
mode is one `marked` generated from `- [ ]` syntax, so no hand-typed `<input>` can masquerade as a
task checkbox.

## What?

When done, **raw HTML embedded in a note renders as literal text** in View mode — `<b>hi</b>` shows
the characters `<b>hi</b>`, not bold text — while **all Markdown/GFM still renders normally**:
headings, emphasis, lists, links, code, fenced code, tables, **task-list checkboxes**, and
**Markdown image syntax** (`![alt](url)`) are unaffected (they are generated by the Markdown parser,
not raw HTML tokens).

**In scope:** a `marked` renderer override in `render.ts` that escapes block- and inline-level raw
HTML tokens; unit tests proving raw HTML is escaped and Markdown features still render; a decisions/doc
note. DOMPurify stays (it still sanitizes URLs/attributes on Markdown-generated elements — e.g. a
`[x](javascript:…)` Markdown link).

**Out of scope:** an allow-list of "safe" HTML tags (all-or-nothing: raw HTML becomes text, full
stop); a per-note toggle for HTML rendering; changing what Markdown syntaxes are supported; touching
the editor.

## Decisions

Locked during specification. Do not relitigate:

- **Escape all raw HTML to text; no tag allow-list.** Rationale: chosen by the user — simplest,
  predictable, and safest. Losing the ability to embed working HTML (`<br>`, `<details>`, `<kbd>`,
  `<sub>`, hand-written HTML tables) is an accepted trade-off; GFM covers the notes use-cases.
- **Implement via a `marked` renderer `html` override, not by pre-escaping the source.** Rationale:
  pre-escaping `<`/`>` in the source would break Markdown that legitimately contains `<` (autolinks
  `<https://…>`) and risks double-escaping. Overriding the renderer's `html` token output escapes only
  the raw-HTML tokens and leaves Markdown-generated HTML intact (verified against `marked` v18:
  `marked.use({ renderer: { html(token){ return escape(token.text) } } })` escapes both block and
  inline raw HTML while task-list checkboxes, tables, and Markdown images still render).
- **Keep DOMPurify.** Rationale: Markdown itself can still produce dangerous output (e.g. a
  `[click](javascript:alert(1))` link renders a real `<a href="javascript:…">`); DOMPurify still needs
  to strip those. Escaping raw HTML and sanitizing are complementary, not redundant.
- **Keep the existing `ADD_ATTR: ["checked","disabled","type"]`.** Rationale: task-list checkboxes are
  Markdown-generated `<input>`s (not raw HTML) and still need those attributes to render.

## How?

### `src/lib/markdown/render.ts`

- Add a small HTML-escape helper (escape `&`, `<`, `>`; escaping `&` first) — or reuse one if a
  suitable util already exists (there is none today; a local `escapeHtml` is fine).
- Register a renderer override alongside the existing options. Current file calls
  `marked.setOptions({ gfm: true, breaks: false })`. Add:
  ```ts
  marked.use({
    renderer: {
      // Raw HTML in a note is untrusted and unwanted — render it as literal text,
      // not as live markup. Markdown-generated HTML (headings, tables, task lists,
      // md images) does NOT go through this hook, so it still renders.
      html(token) {
        const raw = typeof token === "string" ? token : token.text;
        return escapeHtml(raw);
      },
    },
  });
  ```
  (Handle both the token-object and legacy string shapes for forward/backward compatibility across
  `marked` minor versions.)
- Leave `renderMarkdown`'s DOMPurify call unchanged (profiles + `ADD_ATTR`).

### Docs

- `docs/decisions.md` → Product & storage table: add a row — raw HTML in notes is rendered as literal
  text (escaped in the `marked` renderer), with DOMPurify retained for Markdown-generated output.
- `docs/overview.md` → clarify that View renders GFM and shows embedded HTML as plain text.
- `docs/roadmap/index.md` → remove this item from Deferred.

## Caveats

- **Do not regress task lists, tables, or Markdown images.** These are generated by `marked`, not raw
  HTML tokens, so the `html` override must not touch them. The existing "renders task lists as
  checkboxes" test in `tests/markdown/render.spec.ts` must stay green.
- **DOMPurify is still required.** Removing it because "HTML is now escaped" would be wrong — Markdown
  can still emit `javascript:` URLs and other unsafe attributes. Keep the `drops javascript: URLs` and
  event-handler tests green.
- **Escape order matters:** escape `&` before `<`/`>` or you double-encode entities.
- **`marked` token shape.** `marked` v18 passes a token object to `renderer.html` (`{ text, raw, … }`);
  older/newer versions may pass a string. Handle both, or a version bump silently renders `[object
  Object]`.
- **Backwards-compat for existing notes.** A user who previously relied on embedded HTML rendering will
  now see the tags as text. This is intended (see Decisions) and non-destructive — the note *content*
  is unchanged, only how it displays. Worth a line in the PR description.

## Relevant tests

Pure render logic is unit-tested (`tests/markdown/render.spec.ts`).

**Extend `tests/markdown/render.spec.ts`:**
- Raw **inline** HTML is escaped: `renderMarkdown("a <b>x</b> b")` contains `&lt;b&gt;` and **not**
  `<b>`.
- Raw **block** HTML is escaped: `renderMarkdown("<div>x</div>")` contains `&lt;div&gt;` and not
  `<div>`.
- A raw `<input type="checkbox">` typed into a note is escaped (shows as text), **but** a Markdown task
  list (`- [ ] todo`) still renders a real `type="checkbox"` input (guards the desync fix for task
  lists).
- Markdown features still render: headings, tables, fenced code, and **Markdown image** syntax
  `![a](https://e/x.png)` → a real `<img src=…>`.
- Existing sanitization tests still pass: `<script>` stripped, `on*` handlers stripped, `javascript:`
  Markdown links dropped.

**Commands (all must be green):**
```
npm test          # Vitest — extended render spec + existing suite
npm run check
npm run lint
npm run build
```

**Manual E2E (load `dist/` unpacked):**
1. Type `<b>bold?</b>` and `<img src=x onerror=alert(1)>` in a note → View mode shows the literal
   characters, no bold, no image, no alert.
2. A `- [x] done` / `- [ ] todo` task list still renders as checkboxes; a `| A | B |` table still
   renders; `![img](<some data: or https url>)` Markdown image still shows.
3. A `[click](javascript:alert(1))` Markdown link renders as a link with the `javascript:` URL
   stripped (no alert).
